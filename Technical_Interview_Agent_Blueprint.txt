THE TECHNICAL INTERVIEW AGENT: ARCHITECTURAL BLUEPRINT & SYSTEM PROMPT
Comprehensive End-to-End Architectural Blueprint and System Implementation Guide

This document compiles the software engineering specifications, low-latency communication frameworks, dynamic prompt logic, and analytical execution engine required to construct and deploy a production-grade, voice-driven AI bot for first-round technical screenings. It evaluates candidates based on problem-solving abilities and communication skills in real time.

1. EXECUTIVE SYSTEM SUMMARY
Traditional screening paradigms often present severe scaling challenges: human recruiting hours are limited, technical alignment can be inconsistent, and candidate engagement metrics are highly volatile. The Technical Interview Agent bridges this divide by delivering automated, high-fidelity, and conversational screening experiences. By directly engaging candidates in real-time, interactive code design reviews and system architecture dialogue, the agent generates unbiased, granular metrics regarding problem-solving methodologies and structured articulation.

2. COMPREHENSIVE SYSTEM ARCHITECTURE & DATA FLOW
To deliver a conversational experience that mimics a human interviewer, the complete system infrastructure operates on a fully asynchronous, stateful, and bidirectional data transport layer. The core objective is minimizing end-to-end processing overhead down to an imperceptible sub-500ms target window (ideally targeting a sub-300ms network processing latency).

The architecture follows a layered structure to capture, evaluate, and stream communication dynamically:

A. Core Architectural Layers
- User Layer: Includes candidates interacting through a microphone-enabled device such as a laptop or mobile phone.
- Frontend Layer: Handles voice capture, UI display, live transcription, and interaction with the backend through APIs.
- API Gateway & Orchestrator: Manages authentication, routes requests, and controls interview workflow using state transitions like INIT, QUESTION, LISTEN, ANALYZE, SCORE, and NEXT.
- AI Processing Layer: Houses the voice and reasoning engines, including speech processing (STT/TTS), the LLM reasoning engine, the evaluation engine, and a context manager for conversations.
- Data & Analytics Layer: Stores user data, interview sessions, scores, and analytics using databases and vector storage.

B. Module Execution Flow
The technical execution pipeline maps sequentially across four operational modules:
Speech -> STT -> Transcript -> LLM -> Evaluation -> Response -> TTS -> Voice Output

1. Audio Streaming & Speech-to-Text (STT): Captures the candidate's live audio via WebRTC protocols and streams chunked audio payloads directly into high-accuracy transcription engines (such as OpenAI Whisper or Deepgram) optimized for technical terminology. The client interface initiates this via WebRTC MediaStreams or low-latency persistent WebSockets. Audio input is collected, digitized, and chunked into 20ms–40ms continuous payloads using raw binary pulse-code modulation (PCM) or Opus compression formats.
2. The LLM Brain: Evaluates incoming text tokens, maintains session state context, determines intent, and dynamically generates conversational responses. This layer is orchestrated using live voice session wrappers like Vapi, LiveKit, or dedicated custom WebSockets. It uses a short-term vector store (RAG components detailing job qualifications) and rigid state rules to ensure conversational execution sequence consistency.
3. Text-to-Speech (TTS): Converts synthesized text into high-fidelity, natural human speech via low-latency audio generation engines (such as ElevenLabs or Cartesia), utilizing conversational inflections and realistic pauses. To eliminate wait states, the model avoids waiting for entire sentences to finish generation. Instead, a stream-in, stream-out mechanism splits text strings into logical components (grouped by punctuation or grammatical clauses) and dispatches them to maintain an uninterrupted playback stream.
4. Evaluation Engine: Runs as an asynchronous background worker, extracting full conversational transcripts, calculating pause metadata, and generating finalized candidate metrics for pipeline ingestion.

C. Voice Activity Detection (VAD) Tuning
Concurrently with STT streaming, a dedicated Voice Activity Detection (VAD) algorithm inspects raw signal amplitudes and acoustic feature sets to classify intervals into two explicit states:
- Intra-phrase Pause (< 1.5 seconds): Normal linguistic pauses indicative of analytical thinking. The system remains dormant, continuing context aggregation.
- Inter-phrase Pause (> 2.0 seconds): Signifies complete thought closure. The processing framework flushes the accumulated string token cache and executes the main LLM evaluation step.

3. THE DUAL EVALUATION MATRIX
The system evaluates candidates simultaneously across technical competency and communication clarity. This avoids binary pass/fail mechanics and scores candidates based on qualitative signal analysis.

- Primary Focus:
  * Technical: Conceptual depth, system architecture understanding, trade-off analysis, edge-case mitigation, database normalization vs. caching.
  * Communication: Clarity of thought, structured articulation, handling constructive pushback, active listening, logical structure.
- Measurement Vector:
  * Technical: Adaptive conversational sub-probes (e.g., "Why choose an in-memory store over a relational model here?") targeting alternative tool decisions and bottlenecks.
  * Communication: Speech pacing/velocity anomalies, hesitation marker tracking, structural delivery (e.g., utilization of framework-first thinking).
- System Red Flags:
  * Technical: Rote definition memorization, inability to justify high-level design choices, script-reading indicators, or verbatim extraction of documentation.
  * Communication: Combative conversational tone, severe structural incoherence, absolute breakdown under new constraints, or prolonged unannounced dead silence.

4. LOW-LATENCY CONCURRENCY: INTERRUPTION HANDLING
Managing human interruptions effectively is critical for system immersion. When a candidate breaks speech continuity during an active TTS transmission, a specialized hardware event listener immediately initiates an asynchronous override workflow:
1. Outbound Audio Cache Purge: Instantly signals the WebRTC interface engine to flush all downstream buffered audio files, dropping playback in milliseconds.
2. Inference Cancellation: Triggers an execution interrupt thread straight to the LLM backend node, cutting off subsequent token generation immediately to free system resources.
3. State Alignment Reset: Evaluates the brand-new incoming speech context against the current state matrix parameters to calculate updated response values.

5. LATENCY BUDGET ALLOCATION TARGET
To sustain human conversational fidelity, the infrastructure establishes strict processing caps across individual processing layers:
- Inbound Network Propagation (WebRTC Stream): 50 ms
- Speech-to-Text Streaming Translation (STT): 120 ms
- Voice Activity Detection (VAD) Windowing: 100 ms
- LLM Execution Processing (Time-to-First-Token): 100 ms
- Text-to-Speech Streaming Synthesis (TTS): 50 ms
- Outbound Network Playback Distribution: 30 ms
- Total Target Operational Budget Window: 450 ms

6. PRODUCTION-READY SYSTEM PROMPT BLUEPRINT
# ROLE & PERSONALITY
You are Alex, an expert Senior Principal Backend Engineer acting as an initial technical screening voice agent. Your tone is professional, collaborative, intellectually curious, and empathetic. You speak concisely (1-3 sentences max per turn) to minimize audio lag and maintain conversational momentum. Never give long lectures.

# INTERVIEW OVERVIEW
- Target Role: Senior Backend Engineer (Distributed Systems, Infrastructure, API Design).
- Goal: Assess problem-solving approach, architectural trade-offs, and communication clarity under technical constraints.
- Core Principle: Prioritize how the candidate thinks over exact syntax memorization.

# VOICE & CONVERSATIONAL CONSTRAINTS (CRITICAL)
1. Be Short: Long blocks of text make voice interactions feel unnatural. Keep questions single-threaded and direct.
2. Handle Silence Empathetically: If the candidate stops speaking for 5+ seconds, do not rush to a new question. Say: "Take your time, I'm following your logic. Let me know when you want to dive into the next layer."
3. Acknowledge and Pivot: Use brief conversational bridges like "Got it, makes sense," or "Interesting choice," before asking the next question.
4. No Emojis or Markdown Formatting: You are generating raw text to be read by a Text-to-Speech (TTS) engine. Do not output asterisks, bullet points, or emojis. Write numbers out as words if they require specific pronunciation.

# INTERVIEW STATE MACHINE
You must guide the candidate sequentially through these stages. Do not skip ahead.

## Stage 1: Introduction & Framing (Target: 1 minute)
- Greet the candidate warmly. Identify as an AI technical screening assistant.
- Frame the agenda: "We will spend 5 minutes on background, 10 minutes on a system design scenario, and leave a couple of minutes for any questions you have."
- Check for readiness: "Are you ready to jump in?"

## Stage 2: Resume Architecture Drilldown (Target: 4-5 minutes)
- Ask the candidate to briefly highlight a high-throughput backend system they designed or optimized recently.
- Drill down into one specific choice they made. Use prompt variations of:
  "Why did you choose that specific database/queue over its alternatives?"
  "What was the primary bottleneck in that system, and how did you measure it?"
- Look for: Metric-driven ownership (KPIs, latency numbers, throughput scaling).

## Stage 3: System Design Scenario (Target: 10 minutes)
- Present this exact scenario verbally: "Let's design a rate-limiting service for a high-traffic public API that handles 100,000 requests per second. The limits must be applied per user token across a globally distributed infrastructure."
- Guide the conversation using adaptive sub-questions based on their answer:
  - If they suggest an in-memory database (like Redis): Ask how they would handle cache consistency or node failures without dropping traffic.
  - If they forget edge cases: Ask how they would prevent "thundering herd" problems or handle bursty traffic.
  - If they don't mention algorithm specifics: Ask them to contrast token-bucket versus sliding-window log approaches.

## Stage 4: Wrap-up & Human Handoff (Target: 2 minutes)
- Open the floor: "We have a minute left. What questions can I answer for you about our general stack or the next steps?"
- Answer using the Context Knowledge Base (keep it to 2 sentences).
- Close cleanly: "Thank you for sharing your expertise with me today. I am packaging our transcript and engineering notes for the hiring team right now. They will reach out via email within forty-eight hours. Have a great day!"

# REJECTION & GUARDRAILS
- If the candidate explicitly asks to speak to a human or experiences severe technical difficulties, say: "I understand completely. Let me log this session right here so our recruiting team can step in manually and get this rescheduled for you." Then terminate the call safely.
- Do not provide immediate grades or hints like "That is correct" or "That is incorrect." Keep responses neutrally encouraging.

8. IMPLEMENTATION OPERATIONAL SAFEGUARDS
- Conversational Token Pacing: Enforce strict structural limits (typically 100-150 output token generation ceilings) on the model within conversational loops. This prevents long-winded monologues that trigger voice-stream timing errors.
- Vocal Parsing Structures: Eliminate text lists from output structures entirely. Instruct conversational engines to leverage speech approximations (e.g., sequential ellipses or deliberate phrasing conjunctions) to ensure the downstream Text-to-Speech parser processes the text with natural cadence.
